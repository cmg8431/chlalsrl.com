블로그를 새로 만들었다.
"블로그 해야 하는데"를 입에 달고 산 지 몇 년이 지났으니 체감상 몇백 년 만이다.
글을 쓰려고 시작한 일인데, 정신 차려 보니 몇 주째 글은 안 쓰고 사이트만 만지고 있었다.

변명을 하자면 만드는 게 재미있었다.
토스트도, 제스처도, 댓글도 만들다 보니 전부 직접 구현하게 됐다.
좋은 라이브러리가 없어서 그런 건 아니고, 하나씩 붙이다 보니 쓸 이유가 계속 없어졌다.

그 몇 주 동안 밟은 것들을 코드와 함께 남겨둔다.
반나절을 날린 CSS 버그 하나, 3분 만에 버려진 마스코트 하나, 이 도메인이 왜 chlalsrl.com인지도 나온다.
화면들은 배포된 사이트를 헤드리스 브라우저로 녹화했다.

## 토스트 대신 캡슐 하나

화면 구석에 토스트가 쌓이는 UI를 좋아한 적이 없다.
링크 복사, 테마 변경, 댓글 등록. 알릴 건 많은데 전부 다른 자리에서 튀어나오면 정신이 없다.
그래서 상단의 내비게이션 캡슐이 알림까지 겸하게 했다.
평소엔 홈, 검색, 테마, 언어 버튼이 들어 있다.
알릴 게 생기면 컨트롤을 숨기고 메시지 폭만큼 늘어났다가 돌아온다.

![테마를 전환하면 캡슐이 늘어나며 메시지를 보여준다](/posts/blog-details/island.gif)

만들자마자 벽에 부딪혔다.
CSS는 폭을 `auto`로 되돌리는 전환을 애니메이션하지 못하고, 메시지 폭은 내용마다 달라서 목표 폭을 미리 알 수도 없다.
결국 FLIP 방식을 썼다.
현재 폭을 픽셀로 고정해두고 다음 프레임에 목표 폭을 지정해서 트랜지션을 태우는 식이다.

```tsx
const target = message
  ? msg.offsetWidth + PADDING * 2
  : controls.offsetWidth + PADDING;

island.style.width = `${island.offsetWidth}px`;
requestAnimationFrame(() => {
  island.style.width = `${target}px`;
});
```

접힌 뒤에는 폭을 다시 `auto`로 되돌려야 한다.
페이지에 따라 홈 버튼이 생기거나 빠져서 컨트롤 폭이 달라지는데, 고정 픽셀을 남겨두면 여기서 어긋난다.
이걸 모르고 방치했다가, 홈에서만 캡슐이 옆으로 삐져나오는 걸 한참 뒤에 발견했다.

상태 관리는 리스너 Set 하나로 만든 스토어를 `useSyncExternalStore`로 구독한다.
전부 합쳐 70줄이 안 된다.
잠깐 떴다 사라지는 메시지와, 페이지에 머무는 동안 유지되는 컨텍스트(읽기 진행률 같은 것)를 다른 필드로 분리한 게 설계의 전부다.
어느 컴포넌트든 `islandStore.notify()` 한 줄이면 되니, 기능을 추가할 때마다 알림 UI를 고민할 일이 없어졌다.

기능을 다 만들고 나서는 오히려 자잘한 규칙을 정하는 데 시간을 더 썼다.

- 스크롤을 내리면 숨고 올리면 나타난다. 6px 이하의 이동은 무시해서 미세한 스크롤에 떨리지 않게 했다
- 메시지가 떠 있는 동안은 스크롤해도 숨지 않는다
- 캡슐은 `overflow` 클리핑이 있는 래퍼 바깥에 둔다. 클리핑 조상이 있으면 `backdrop-filter`가 깨진다

서두에서 말한 반나절짜리 버그가 마지막 항목이다.
어느 날 캡슐의 블러가 통째로 사라졌는데, 캡슐 쪽 코드는 아무리 봐도 멀쩡했다.
범인은 레이아웃 정리하면서 래퍼에 넣은 `overflow` 클리핑 한 줄.
버그는 항상 내가 안 보고 있는 파일에 있다.

## 페이지 전환

웹은 페이지가 바뀔 때 한 번 뚝 끊긴다.
네이티브 앱과 비교해서 늘 아쉬웠던 부분인데, 이제는 View Transitions API라는 표준이 그 틈을 메꾼다.
Next.js와는 next-view-transitions로 연결했다.
브라우저 내부 동작이라 직접 만들 영역도 아니니 이건 순순히 라이브러리를 썼다.

![홈에서 글 목록, 글 상세로 이동하는 화면 전환](/posts/blog-details/view-transition.gif)

홈의 이름 워드마크에 `view-transition-name`을 붙여두면, 페이지가 바뀔 때 브라우저가 이전 화면과 다음 화면의 스냅샷을 잡아 그 요소만 이어서 보간한다.
라우팅 코드는 평범한 링크 그대로다.
미지원 브라우저에서는 즉시 전환으로 떨어진다.
이런 기능은 안 되는 환경에서 조용히 빠지는 게 맞다고 본다.

## 모바일에서 글 넘기기

모바일에서 글을 다 읽으면 목록으로 돌아가서 다음 글을 찾아야 한다.
이 동선이 예전부터 마음에 안 들었다.
책장 넘기듯 옆으로 넘어가면 좋겠다 싶어서 만들었는데, 결과적으로 이번 개편에서 가장 오래 붙잡고 있었던 인터랙션이 됐다.

글을 읽다가 화면을 옆으로 당기면 가장자리에서 오버레이가 나오면서 이전 글 제목이 보인다.
충분히 당겨서 놓으면 이동, 어중간하면 스냅백.

![글 상세를 오른쪽으로 당기면 이전 글 제목이 나타나고, 놓으면 슬라이드 전환으로 이동한다](/posts/blog-details/swipe.gif)

제스처 라이브러리 없이 touchstart, touchmove, touchend 세 개로 만들었다.
이벤트 처리 자체는 별게 없다.
어려운 건 판정이었다.

- 가로 이동이 10px을 넘고, 세로 이동의 1.2배보다 클 때만 스와이프로 인정한다
- 판정 전에 세로로 12px 이상 움직이면 그 터치는 끝까지 무시한다. 스크롤 의도로 본다
- 코드블록, 표처럼 자체 가로 스크롤이 있는 요소에서 시작한 터치도 무시한다
- 일단 인정되면 그때부터 `preventDefault()`로 스크롤을 잠근다. 리스너는 `passive: false`로 걸어야 한다

당기는 감각은 감쇠 함수 하나가 만든다.

```ts
const THRESHOLD = 88;
const damp = (p: number) =>
  0.32 * Math.min(p, THRESHOLD) + 0.14 * Math.max(0, p - THRESHOLD);
```

손가락이 움직인 거리를 그대로 쓰면 본문이 너무 많이 밀려서 싸 보인다.
임계점까지는 0.32배, 그 뒤로는 0.14배로 눌러야 저항감이 생긴다.
88px을 넘는 순간 오버레이 톤을 살짝 올려서 "여기서 놓으면 이동"이라는 상태를 알린다.
계수들에 근거는 없다.
실제 기기에서 손으로 당겨보면서 맞춘 값이다.
이런 값을 정하는 데 손 말고 다른 방법이 있는지는 아직도 모르겠다.

놓았을 때는 오버레이를 걷으면서 View Transitions의 슬라이드 전환으로 넘긴다.
당기던 방향 그대로 페이지가 밀려나가므로 제스처와 전환이 한 동작으로 이어진다.
이동 가능한 글은 미리 `router.prefetch`를 해둬서 놓는 순간 기다림이 없다.
스냅백은 `cubic-bezier(0.34, 1.3, 0.5, 1)`로, 살짝 넘었다 돌아오는 스프링 곡선이다.

보이는 쪽도 순탄하지 않았다.
첫 버전은 원형 화살표 칩에 모노 폰트 라벨 카드, 앰버 색 강조까지 들어간 요란한 프리뷰였다.
두 번 갈아엎고 나니 그라데이션 오버레이와 글 제목만 남았다.
이것저것 얹을수록 오히려 정신없어 보여서, 결국 다 덜어냈다.

오버레이는 `createPortal`로 body에 직접 그린다.
본문에 `transform`이 걸리는 순간 그 안의 `position: fixed`는 뷰포트 대신 본문을 기준 삼기 때문이다.
스와이프 중에는 본문 전체가 transform으로 밀리고 있으니, 오버레이를 본문 안에 두면 같이 밀려버린다.

여담으로, 위 녹화를 만들 때 처음엔 왼쪽으로 당겼는데 아무 일도 일어나지 않았다.
버그인 줄 알고 코드를 열었는데 버그가 아니었다.
최신 글이라 다음 글이 없었던 것뿐이다.
이동할 대상이 없으면 제스처 자체를 거부하도록 짜여 있었으니, 내가 만든 방어 로직에 내가 막힌 셈이다.

## 문장 하나를 공유하기

글 링크를 통째로 공유받는 것보다, 누가 내 글의 한 문장을 긁어서 보내주는 쪽이 훨씬 기분 좋다.
그 문장이 그 사람한테 남았다는 뜻이니까.
본문에서 텍스트를 선택하면 공유 버튼이 뜨는 건 그래서다.

![본문에서 문장을 선택하면 공유 버튼이 뜨고, 누르면 문장 링크가 복사된다](/posts/blog-details/highlight.gif)

이 기능에서 서버가 하는 일은 없다.
URL 표준이 거의 다 해준다.

```ts
const url = `${location.origin}${location.pathname}#:~:text=${encodeURIComponent(clean)}`;
```

Text Fragment(`#:~:text=`)는 브라우저가 네이티브로 지원하는 문법이라, 링크를 연 쪽에서 알아서 해당 문장까지 스크롤하고 하이라이트까지 해준다.
앵커를 저장할 DB도, 본문에 심을 ID도 필요 없다.
내 코드가 하는 일은 `selectionchange`를 듣다가 조건에 맞으면 버튼을 띄우는 것뿐이다.

- 8자 미만은 무시한다. 실수로 긁은 것에 가깝다
- 300자 초과도 무시한다. 그건 문장 공유가 아니라 본문 복사다
- 본문 영역 밖의 선택은 무시한다

버튼에는 `onMouseDown={(e) => e.preventDefault()}`가 걸려 있다.
처음엔 이게 없었는데, 버튼을 누르는 순간 선택이 풀리면서 버튼도 같이 사라졌다.
클릭이 안 되는 게 아니라 클릭하기 전에 버튼이 없어지는 거라, 원인을 이해하는 데 시간이 걸렸다.

공유된 문장은 Supabase에 슬러그와 함께 집계된다.
여러 명이 같은 문장을 공유하면 글 하단에 "독자들이 밑줄 그은 문장"으로 노출된다.
어떤 문장이 남을지는 나도 궁금하다.

## 읽기 설정과 깜빡임

다크 모드에서 새로고침하면 흰 화면이 번쩍이는 문제는 유명하다.
그런데 글자 크기 설정도 정확히 같은 문제를 일으킨다는 얘기는 별로 없더라.

글 상세에는 글자 크기 3단과 집중 모드가 있다.
설정값은 `<html>`의 데이터 속성으로 적용해서 localStorage에 저장하는데, 정작 이 기능에서 제일 중요한 코드는 React 컴포넌트가 아니라 그보다 먼저 실행되는 인라인 스크립트다.

```js
(function () {
  try {
    var d = document.documentElement, t = localStorage.getItem("theme");
    if (t === "dark" || t === "light") d.dataset.theme = t;
    var s = localStorage.getItem("reading-size");
    if (s === "sm" || s === "lg") d.dataset.readingSize = s;
  } catch (e) {}
})();
```

저장값 반영이 hydration 이후로 밀리면, 글자를 lg로 설정한 사람은 새로고침마다 글자가 작았다 커지는 걸 보게 된다.
테마 번쩍임과 원인이 같으니 해법도 같다.
테마든 글자 크기든, 첫 페인트 전에 같은 스크립트에서 전부 심는다.

집중 모드는 뷰포트 세로 중앙 60% 밴드에 걸친 문단만 남기고 나머지를 가라앉힌다.
긴 글을 읽다 보면 눈이 자꾸 다른 줄로 새는 편이라, 나부터 쓰려고 넣은 기능이다.

![집중 모드를 켜면 화면 중앙 부근의 문단만 밝게 남는다](/posts/blog-details/focus-mode.gif)

스크롤마다 본문 블록들의 `getBoundingClientRect`를 재서 밴드에 걸치는지로 클래스를 토글한다.
계산은 `requestAnimationFrame`으로 스로틀해서 프레임당 한 번만 돈다.
글 하나의 블록 수십 개 수준에서는 이걸로 충분해서, IntersectionObserver로 바꿀 이유를 아직 못 찾았다.

한 가지 보정이 있다.
밴드보다 큰 블록이 걸리면 조건식만으로는 화면 전체가 어두워지는 순간이 생긴다.
그래서 중앙에 가장 가까운 블록 하나는 조건과 무관하게 항상 밝힌다.
긴 문단 한가운데서 스크롤을 멈춰보고 나서야 필요한 걸 알았다.

## 백엔드 없는 댓글

로그인을 요구하는 순간 개인 블로그의 댓글은 죽는다고 생각한다.
그렇다고 익명으로 열어두면 이름 칸이 문제다.
뭘 쓸지 고민하다 그냥 창을 닫게 되니까.

이 블로그의 댓글 폼은 처음부터 랜덤 닉네임이 채워진 채 열린다.
옆의 버튼을 누르면 70ms 간격으로 여덟 번 이름이 바뀌다 멈춘다.
마음에 드는 이름이 나올 때까지 돌리면 된다.

![이름 뽑기 버튼을 누르면 닉네임이 슬롯처럼 돌아가다 멈춘다](/posts/blog-details/nickname-roll.gif)

아바타는 저장하지 않는다.
이름 문자열의 해시로 이모지 16종과 배경색 6종에서 골라 그리기 때문에, 같은 이름은 언제나 같은 아바타가 되고 이름이 바뀌면 아바타도 같이 바뀐다.
슬롯이 돌아가는 동안 아바타까지 함께 도는 건 이 구조의 부수 효과인데, 결과적으로 제일 마음에 드는 부분이 됐다.
의도한 것보다 잘 나온 기능이 가끔 있다.

백엔드는 따로 없다.
브라우저에서 Supabase REST를 직접 호출하고 권한은 RLS에 맡겼다.
환경변수가 없는 로컬에서는 localStorage에만 저장되는 폴백을 뒀는데, 덕분에 오프라인에서도 개발이 된다.
스팸 방어는 화면 밖에 숨긴 honeypot 입력 하나로 시작했다.
뚫리면 그때 다음 수를 두면 된다.

기능보다 오래 걸린 건 입력 카드의 모양이었다.
내부 구분선을 지우고, 포커스 링을 넣었다가 색이 튀어서 중립 톤으로 되돌리고, 결국 박스 링 대신 밑선으로 바꿨다.
마지막엔 전역 `focus-visible` 아웃라인이 텍스트 입력에도 걸려서 카드 안에 선이 겹치길래 입력 요소만 예외로 뺐다.
댓글 카드 하나에 스타일 커밋이 네 개 쌓였다.
디자이너 없이 개발하면 이렇게 된다.

## 검색창에 숨긴 것

⌘K를 누르면 검색 팔레트가 뜬다.
글 제목 검색과 최근 본 글까지는 평범한데, 명령어를 치면 터미널처럼 동작한다.
`help`, `theme dark`, `go blog`, `whoami`, 그리고 `qwerty`.

![⌘K 팔레트에 qwerty 최민기를 입력하면 chlalsrl이 출력된다](/posts/blog-details/command-menu.gif)

서두에서 예고한 도메인의 비밀이 여기 있다.
`qwerty`는 한글을 쿼티 자판 그대로의 영문으로 바꾸는 명령인데, `qwerty 최민기`를 치면 `chlalsrl`이 나온다.
한영키를 안 누른 채 "최민기"를 치면 그대로 chlalsrl.
도메인을 불러줄 때마다 설명이 길어지는 게 단점이지만, 설명을 다 들은 사람은 한 번에 기억한다.

방향키로 코나미 커맨드를 입력하면 상단 캡슐이 반겨준다.
개발자 도구 콘솔에는 배포된 커밋 해시가 찍혀 있다.
아무도 못 찾아도 상관없는 것들이다.
그래도 언젠가 누가 ⌘K에 `sudo`를 쳐보고 웃는다면, 만든 보람은 충분하다.

## 두 눈

사이트의 아이콘은 어두운 배경에 떠 있는 두 눈이다.
파비콘부터 홈 화면 아이콘까지 전부 이 얼굴인데, 이미지 파일 없이 코드로 그린다.

원안은 SVG로 한 번 그려두고 그 비율을 상수로 옮겼다.
눈 크기는 캔버스의 0.242배, 동공은 눈의 0.477배 하는 식이다.
덕분에 64px 파비콘부터 512px 홈 화면 아이콘까지 같은 코드에서 나온다.
512px은 maskable이라 모서리를 OS가 마음대로 깎는데, 눈을 세이프존 안쪽에 뒀기 때문에 어떤 모양으로 잘려도 얼굴은 남는다.
글자가 없는 디자인이라 아이콘 생성에 폰트를 로딩할 필요도 없어졌다.

공유 카드에도 같은 정체성을 이으려고 했다.
OG 이미지는 글마다 제목, 날짜, 태그를 읽어 1200×630으로 빌드 타임에 생성된다.

![자동 생성되는 오픈 그래프 이미지](/posts/blog-details/og-image.png)

이 카드도 세 번째 버전인데, 서두에서 말한 3분짜리 마스코트가 여기서 나온다.
이번 개편에서 두 눈을 카드에도 넣었다가, 커밋 기록 기준 정확히 3분 뒤에 뺐다.
1200×630 안에서는 제목이 주인공이어야 하는데 눈이 자꾸 시선을 가져갔다.
넣을 때는 몰랐는데 빼고 나니 바로 보였다.
지금은 배지의 작은 글로우 도트 하나로만 흔적을 남겼다.

## 그 외에 밟은 것들

본문에 담기 애매한 삽질들.

- **등장 애니메이션은 순수 CSS다.** 처음엔 IntersectionObserver로 JS에서 게이팅했는데, 느린 기기에서 첫 페인트와 애니메이션 시작이 어긋나 모션이 두 번 보였다. CSS 애니메이션과 `animation-delay`로 바꾸니 사라졌다
- **한글 태그 페이지가 404였다.** `generateStaticParams`에서 태그를 미리 `encodeURIComponent`로 인코딩해 넘겼더니, Next가 한 번 더 인코딩해서 이중 인코딩된 경로로 프리렌더됐다. 원본을 그대로 넘기니 해결됐다
- **다크 모드에서 드래그하면 화면이 번쩍였다.** 기본 반전 셀렉션이 어두운 화면에서 넓은 흰 면으로 뒤집히는 게 원인이었다. 다크에서는 액센트 색을 26%만 섞은 면과 밝은 글자로 바꿔서 완화했다
- **본문 폰트는 두 벌이다.** 라틴과 숫자는 Geist, 한글은 Pretendard Variable을 다이내믹 서브셋으로 얹었다. 폰트 스택 순서만 맞추면 한 문장 안에서 알아서 섞인다

사이트는 한국어, 영어, 일본어를 지원하고, 번역이 없는 글은 요청 언어에서 영어, 한국어 순으로 폴백해 원문을 보여준다.

## 정리

라이브러리를 쓰지 말자는 주의는 아니다.
View Transitions처럼 직접 만들 수 없는 건 라이브러리를 썼고 앞으로도 그럴 것이다.
다만 이번에 직접 만든 것들은 하나같이 코드가 200줄을 넘지 않는다.
의존성이었다면 설정과 씨름했을 시간에, 동작을 원하는 만큼 다듬었다.

시대 덕도 봤다.
요즘은 AI와 같이 짜면 이 정도 규모의 구현은 금방 나온다.
라이브러리 문서를 읽는 시간과 직접 만드는 시간이 비슷해졌으니, 그럴 바엔 내 손에 맞는 쪽을 고르게 된다.
블로그 정도의 규모라면 이 트레이드오프는 꽤 남는 장사였다.

남은 일도 있다.
이력서 페이지는 아직 작업 중이고, 밑줄 그은 문장이 쌓이면 보여줄 방식도 다듬어야 한다.
그래도 사이트 핑계는 이제 못 댄다.
몇백 년 만에 만들었으니, 이제 글을 쓸 차례다.
